# Файл: 01_общий_аудит_кода.md

## 1. Методология и охват

### 1.1 Что это за сайт и фактический продукт
Сайт `Whale Wzrd` — русскоязычный performance/lead-generation landing и контентная площадка специалиста/агентства по Google Ads, Meta Ads, консультациям и рекламной аналитике. В кодовой базе есть главная посадочная страница, сервисные страницы `meta-ads`, `google-ads`, `consult`, `meta-apps`, блог/кейсы, FAQ, маркетинговый глоссарий, калькуляторы бюджета/ROI, формы заявок, thank-you flow, legal pages, админ-раздел статей, Cloudflare Pages Functions API, D1/R2/KV-интеграции, JSONBin fallback и Meta CAPI диагностика. Поэтому аудит рассматривает сайт не как статичную визитку, а как коммерческую SPA/SSG-посадочную систему с контентом, CRM-like lead capture, рекламными пикселями и серверными conversion API.

### 1.2 Уточнённый Cloudflare production context без раскрытия секретов
В production Cloudflare Pages заданы секреты `ADMIN_PASSWORD`, `JSONBIN_BIN_ID`, `JSONBIN_MASTER_KEY`, `META_CAPI_ACCESS_TOKEN`, `META_CAPI_DEBUG_SECRET`, `META_CAPI_TEST_CODE`, `TRACKING_HMAC_SECRET`; их значения в аудит намеренно не включаются. Также заданы plaintext/build/runtime параметры: `META_CAPI_API_VERSION=v25.0`, `META_OUTBOX_MAX_ATTEMPTS=8`, `R2_PUBLIC_HOST` на `r2.dev`, `REQUIRE_FRESH_ARTICLES=true`, `SITE_URL=https://www.whalewzrd.com`, `TRACKING_SIGNATURE_MODE=monitor`, `TRACKING_SIG_TTL_SEC=60`, `USE_D1_ARTICLES=true`, `VITE_ANALYTICS_RUNTIME=direct`, GA/GTM/Meta/Yandex IDs. Bindings: KV `META_CAPI_DIAGNOSTICS`, `META_CAPI_IDEMPOTENCY`, `META_CAPI_NONCE`, D1 `DB=landing-prod`, R2 `BUCKET=whale-wzrd-uploads`. Это меняет оценку: sitemap canonical через `SITE_URL` в production корректен, CAPI token фактически должен быть доступен, outbox имеет лимит 8 попыток, статьи должны требовать freshness и использовать D1.

Аудит выполнен по фактической кодовой базе React/Vite/Cloudflare Pages Functions. Проверены HTML-точка входа, маршрутизация, CSS-слои, ключевые UI-компоненты, consent/analytics runtime, API-функции, SEO endpoints, security middleware, robots/redirects и сборка. Полный буквальный построчный комментарий всех shadcn/Radix UI файлов занял бы сотни страниц; ниже приведён практический построчный разбор критичных участков и матрица типовых строк по остальным файлам. Для окончательной верификации нужны production HAR, WebPageTest/Lighthouse mobile traces и реальные Cloudflare headers.

## 2. Построчный разбор ключевых файлов

### 2.1 `index.html`
1. `<!DOCTYPE html>` — корректный HTML5 режим. 🟢
2. `<html lang="ru">` — язык задан, хорошо для a11y/SEO. 🟢
3-5. Head и charset корректны; facebook-domain-verification расположен рано и не блокирует. 🟢
6. `viewport width=device-width, initial-scale=1.0` корректен для mobile-first; нет `maximum-scale`, значит pinch zoom не заблокирован. 🟢
7-18. Базовые title/description/OG/Twitter/canonical есть. 🟡 Проблема: React-компонент SEO меняет meta только на клиенте, поэтому crawlers без JS увидят только главную мету.
19. RSS alternate есть. 🟢
20-35. Preconnect/dns-prefetch к картинкам и трекерам. 🟡 На мобильном 3G лишние preconnect к трекерам до consent расходуют сокеты/энергию.
22-29. Preload hero-изображения с `imagesrcset/imagesizes`. 🟡 URL `?width=` на `i.ibb.co` может не давать реального ресайза; проверить фактический вес.
36. Inline critical CSS минимален. 🟢
41. Module script подключён без `defer`, но `type=module` defers по умолчанию. 🟢

### 2.2 `src/main.tsx`
1-4. Импорты минимальны: React root, App, ErrorBoundary, CSS. 🟢
6. `document.getElementById("root")!` использует non-null assertion. 🟡 При сломанном HTML будет runtime crash; лучше явная проверка.
7-9. ErrorBoundary закрывает приложение целиком. 🟢

Пример улучшения:
```ts
const rootNode = document.getElementById('root');
if (!rootNode) throw new Error('Root element #root not found');
createRoot(rootNode).render(<AppErrorBoundary><App /></AppErrorBoundary>);
```

### 2.3 `src/app/App.tsx`
2-4. Router и ArticlesProvider подключены на верхнем уровне. 🟢
8. `<div className="dark">` принудительно включает тёмную тему. 🟡 Нет поддержки `prefers-color-scheme` и выбора пользователя.
9-11. Provider окружает все маршруты. 🟢

### 2.4 `src/app/routes.tsx`
5, 7-16, 21. Lazy loading применён для большинства страниц. 🟢
17-20. `MetaAdsPage`, `GoogleAdsPage`, `ConsultPage`, `MetaAppsPage` импортируются eagerly. 🟡 Эти посадочные страницы попадают в главный бандл и ухудшают мобильный TTI/INP.
24-52. Route error boundary перезагружает страницу при ошибке chunk load. 🟢
63-71. CookieConsentManager lazy + Suspense. 🟢
83-111. Маршруты читаемы. 🟡 `/admin` клиентский маршрут присутствует; robots закрывает, но лучше защитить на сервере и не отдавать админский chunk публично.

Решение:
```ts
const MetaAdsPage = lazy(() => import('./pages/MetaAdsPage'));
// аналогично GoogleAdsPage, ConsultPage, MetaAppsPage
```

### 2.5 `src/styles/index.css`
1-4. CSS разбит на fonts/tailwind/theme/blog-readable. 🟢
7-29. Анимации scanline/glow/scanner. 🟡 Нет глобального `prefers-reduced-motion`; мобильные пользователи со слабым CPU получают лишнюю нагрузку.
31-54. Компонентные стили для legal copy; мобильный fallback `overflow-wrap:anywhere` на <=480px. 🟢

Решение:
```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    scroll-behavior: auto !important;
  }
}
```

## 3. Архитектура
- 🟢 Структура понятна: `src/app/components`, `pages`, `functions/api`, `_lib`, `migrations`, `scripts`.
- 🟡 UI-библиотека содержит много Radix/shadcn компонентов, но часть зависимостей тяжёлая для landing: MUI, Three, TipTap, React DnD, Recharts. Даже при code splitting они увеличивают install/build surface.
- 🟡 Нет отдельного слоя typed API client; формы вызывают `fetch` напрямую, что усложняет retry/offline queue. С учётом Cloudflare `META_OUTBOX_MAX_ATTEMPTS=8` серверная очередь CAPI частично закрывает потери на стороне Meta, но не закрывает потерю самого лида на клиенте до успешного POST.
- 🟢 Cloudflare Functions имеют отдельные `_lib` для http/cache/meta/auth/sanitize.

## 4. Производительность кода
- 🟠 Главный JS chunk после сборки: `index-*.js` 334.96 kB raw / 99.40 kB gzip, `vendor` 228.72 kB raw / 74.95 kB gzip, CSS 207.76 kB raw / 27.67 kB gzip. Для мобильного 3G это риск TTI/INP.
- 🟡 `motion` отдельным chunk 94.89 kB raw; анимации используются активно.
- 🟡 CSS 207 kB raw указывает на широкую Tailwind/Radix поверхность; критический CSS не выделен.
- 🟢 Lazy routes снижают initial route cost, но eager service pages нужно ленивить.
- 🟡 3D-модель `public/models/whale.glb` требует проверки lazy loading, Draco/meshopt и отключения на low-end/mobile.

## 5. Кросс-браузерность и адаптивность
- 🟢 Tailwind breakpoints используются (`sm`, `md`, `lg`), формы имеют мобильные размеры.
- 🟡 Нет документированной breakpoint-матрицы. Нужно зафиксировать 320/360/375/390/414/768/1024/1280.
- 🟡 CSS использует `text-wrap: pretty`; в старых браузерах возможен fallback без эффекта, это не критично.
- 🟡 Не найден единый visual regression набор для iOS Safari/Chrome Android.

## 6. Логирование и ошибки
- 🟢 ErrorBoundary есть на уровне приложения и route chunk errors.
- 🟡 В analytics runtime есть `console.info/warn`; в production это раскрывает runtime config и шумит.
- 🟡 Формы используют `alert()` для ошибок; мобильный UX хуже, нет inline retry.

## 7. Мобильная версия
- 🟢 Viewport корректный, pinch zoom разрешён.
- 🟡 Touch-detection отключает hover-анимации в ContactForm, но не глобально.
- 🟠 Нет явного performance budget под mobile: LCP image, JS <=170 kB gzip, INP <=200 ms.
- 🟡 Нет `srcset/picture` в аудированных React-компонентах; preload есть только в HTML.
- 🟡 Не проверены landscape keyboard resize, safe-area inset, iOS visual viewport.
