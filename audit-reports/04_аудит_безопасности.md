# Файл: 04_аудит_безопасности.md

## 1. Общий вывод
Базовая security-позиция хорошая: Cloudflare middleware ставит CSP, HSTS, nosniff, Referrer-Policy, Permissions-Policy; API лидов проверяет origin/signature/rate limit/honeypot; HTML sanitizer присутствует. С учётом Cloudflare production settings секреты вынесены из репозитория и хранятся как encrypted variables, что является сильной стороной. Главные риски: CSP разрешает `unsafe-inline` для scripts/styles, `frame-ancestors 'self'` и X-Frame-Options SAMEORIGIN допускают self-framing, нет SRI для внешних скриптов (динамически вставляются), admin route публично доступен клиентом, JSON parse может приводить к 500, npm audit не прошёл из-за отсутствия package-lock при npm-аудите (проект pnpm).

## 2. Клиентские уязвимости
- 🟢 React escaping снижает XSS для обычного JSX.
- 🟢 Есть `src/app/utils/sanitizeHtml.ts` и server `_lib/sanitize.ts` (нужно отдельно проверить все `dangerouslySetInnerHTML`).
- 🟡 Динамические meta/JSON-LD используют JSON.stringify, безопаснее raw HTML.
- 🟠 CSP `script-src 'self' 'unsafe-inline' ...` допускает inline XSS. Для современного уровня нужен nonce/hash based CSP.
- 🟡 `appendExternalScript` загружает third-party scripts без SRI; для GTM/GA/Yandex SRI практически сложно из-за изменяемости, но для статичных third-party возможно.

## 3. Кликджекинг
- 🟡 `frame-ancestors 'self'` и `X-Frame-Options: SAMEORIGIN` защищают от внешних сайтов, но разрешают self-framing. Для landing лучше `frame-ancestors 'none'`, если нет легитимного embedded use case.
- 🟡 Mobile viewport clickjacking: нет защиты от overlay внутри своего домена/preview. Нужен review всех modal/sheet z-index.

## 4. API и серверные аспекты
- 🟢 `/api/lead` проверяет trusted request origin и tracking signature mode.
- 🟢 Rate limit и honeypot есть.
- 🟡 `JSON.parse` без try/catch может вызвать 500 на invalid JSON. Вернуть 400.
- 🟡 `DEFAULT_LEAD_ENDPOINT` содержит Google Apps Script URL в коде; если endpoint приватный, перенести в env. В Cloudflare уже есть секреты для JSONBin/Meta/Admin/Tracking, поэтому тот же подход стоит применить и к любым webhook endpoints.
- 🟢 SQL-инъекции в аудированных участках не видны; D1 доступ нужно проверять в `_lib/d1.ts` и admin endpoints.
- 🟡 Admin endpoints нужно проверить на auth в каждом файле, а не только UI route. Наличие encrypted `ADMIN_PASSWORD` в Cloudflare — хороший признак, но аудит должен подтвердить, что все `/api/admin/*` реально вызывают server-side auth.

## 5. Аутентификация и сессии
- 🟡 В видимой части нет пользовательской сессии. Admin route существует; нужно проверить `functions/_lib/auth.ts` и все `/api/admin/*` endpoints.
- 🟡 Consent details в localStorage не httpOnly, это ожидаемо для client consent, но не хранить там raw PII. Сейчас hashed Meta user data хранится, лучше минимизировать TTL и document policy.

## 6. Заголовки безопасности
Текущие headers:
- `X-Content-Type-Options: nosniff` 🟢
- `X-Frame-Options: SAMEORIGIN` 🟡
- `Referrer-Policy: strict-origin-when-cross-origin` 🟢
- `Permissions-Policy: camera=(), microphone=(), geolocation=()` 🟢
- `COOP: same-origin-allow-popups` 🟢/🟡 зависит от Meta popups
- `CSP` 🟠 из-за unsafe-inline
- `HSTS max-age=31536000; includeSubDomains; preload` 🟢

Рекомендуемый CSP-направление:
```http
Content-Security-Policy:
  default-src 'self';
  script-src 'self' 'nonce-{RANDOM}' 'strict-dynamic' https://www.googletagmanager.com ...;
  style-src 'self' 'nonce-{RANDOM}';
  object-src 'none';
  base-uri 'self';
  frame-ancestors 'none';
  upgrade-insecure-requests;
```

## 7. CORS
- 🟡 Нужно проверить `functions/_lib/http.ts`: важно не отдавать `Access-Control-Allow-Origin: *` для authenticated/admin endpoints.
- 🟢 Lead API проверяет trusted origin до обработки.

## 8. Cloudflare secrets, variables и bindings
- 🟢 Секреты `ADMIN_PASSWORD`, `JSONBIN_*`, `META_CAPI_ACCESS_TOKEN`, `META_CAPI_DEBUG_SECRET`, `META_CAPI_TEST_CODE`, `TRACKING_HMAC_SECRET` заданы как encrypted variables; значения не должны попадать в репозиторий, логи, PR и клиентский bundle.
- 🟢 Bindings KV/D1/R2 разделяют diagnostics/idempotency/nonce/articles/uploads: это правильная cloud-native архитектура.
- 🟡 `TRACKING_SIGNATURE_MODE=monitor` снижает риск сломать аналитику, но не блокирует поддельные tracking requests. После анализа false positives рекомендуется `enforce`.
- 🟡 `R2_PUBLIC_HOST` на `r2.dev` удобен, но для production SEO/brand/cache/security лучше рассмотреть custom domain с контролируемыми headers.
- 🟡 `META_CAPI_API_VERSION=v25.0` нужно включить в календарь upgrade-аудита Meta Graph API.
- 🟢 `TRACKING_SIG_TTL_SEC=60` ограничивает replay-window; KV `META_CAPI_NONCE` помогает бороться с повтором событий.

## 9. Зависимости и CVE
- 🟡 `npm audit` не применим напрямую: проект имеет `pnpm-lock.yaml`, npm требует package-lock. Нужно запускать `pnpm audit` или создать временный lock в CI.
- 🟡 Много UI dependencies расширяют supply chain. Включить Dependabot/Renovate и lockfile integrity checks.

## 10. Конфиденциальность
- 🟢 PII для Meta хешируется на server/client.
- 🟡 До согласия внешний geo fallback `ipwho.is` получает IP. Нужна правовая оценка.
- 🟡 Console logs могут раскрывать config, fbtrace_id, errorText.
- 🟡 FZ-152/GDPR: формы собирают имя/email/phone; политика есть, но нужно проверить актуальность текста, операторов, трансграничную передачу Meta/Google/Yandex.

## 11. Инфраструктура
- 🟡 Открытые порты/Cloudflare settings не проверялись: нужен внешний scan, DNS records, TLS test, bucket enumeration.
- 🟢 Canonical redirect есть на middleware и `_redirects`.

## 12. Мобильная безопасность
- 🟡 Deep links Telegram/WhatsApp нужно проверять: нет ли open redirect через user input.
- 🟡 PWA/service worker не обнаружен в аудированных файлах; если будет добавлен, проверять cache poisoning и stale auth.
- 🟡 Mobile WebView: third-party cookies/ITP могут ломать consent/analytics; нужна graceful degradation.
