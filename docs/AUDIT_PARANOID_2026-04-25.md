# Параноидальный аудит (2026-04-25)

## Общая оценка
- Проект функционально рабочий и ориентирован на SEO/AEO (sitemap/feed, schema.org, FAQ), но уровень production-hardening для high-load и security-critical сценариев **недостаточный**.
- Ключевые риски: XSS-контур вокруг пользовательского HTML, неатомарный rate-limit, слабая операционная зрелость (нет test/lint scripts в npm, нет lockfile), нестрогая обработка отказов внешних систем.

## CRITICAL

### C1. Stored XSS в контенте статей (двойной контур рендера)
1) **Где**: `functions/_lib/jsonbin.ts`, `functions/_lib/seo.ts`, `src/app/pages/BlogPage.tsx`, `src/app/components/ArticleEditor.tsx`.
2) **Тип**: Security (Stored XSS) + SEO integrity risk.
3) **Почему**: Санитизация regex-ами удаляет только часть векторов (`<script>`, часть `on*`, `javascript:`), но пропускает сложные payloads (атрибуты без кавычек, data URI, SVG-пейлоады, DOM clobbering и т.д.). Затем HTML рендерится напрямую в SPA и в бот-SSR.
4) **Прод-проявление**: выполнение произвольного JS у пользователей/ботов, перехват данных, подмена контента, SEO poisoning.
5) **Условия**: компрометированный админ-доступ / ошибка редактора / вредоносный импорт статьи.
6) **Критичность**: **CRITICAL**.
7) **Цепочка последствий**:
   - инъекция в JSONBin;
   - кеширование и раздача зараженной страницы;
   - компрометация сессий/бренд-репутации;
   - блокировка поисковиками/браузерами;
   - падение конверсии и инцидент response.
8) **Воспроизведение**: добавить в `rawHtml` блок XSS-вектор, сохранить статью, открыть `/blog/:slug`.
9) **Исправление (минимально инвазивно)**: перейти с regex на whitelist sanitizer (DOMPurify/isomorphic-dompurify) на записи + на чтении; запретить unsafe теги/атрибуты; для ссылок enforce `rel="noopener noreferrer"`.
10) **Альтернативы**: хранить структурированные блоки вместо raw HTML и рендерить только безопасные компоненты.

## HIGH

### H1. Rate-limit неатомарен и уязвим к гонкам/обходу
1) **Где**: `functions/_lib/rate-limit.ts`, `functions/_lib/http.ts`.
2) **Тип**: Reliability/Security (DoS, brute-force hardening).
3) **Почему**: паттерн `match -> increment -> put` в cache не атомарен; параллельные запросы перетирают счетчик. Дополнительно fallback на `x-forwarded-for` может быть спуфнут вне доверенной цепочки.
4) **Прод-проявление**: brute-force и burst-трафик проходят лимит, защита админ-эндпойнтов деградирует.
5) **Условия**: конкурентные запросы, распределенный трафик по PoP.
6) **Критичность**: **HIGH**.
7) **Цепочка**: burst -> рассинхрон счетчиков -> больше 30 rpm фактически -> рост 401/PUT нагрузки -> деградация.
8) **Воспроизведение**: отправить 100 параллельных POST/PUT к `/api/admin/articles` с одного IP, сравнить ожидание vs факт.
9) **Исправление**: Cloudflare Durable Objects/KV+DO для атомарности, sliding window/token bucket.
10) **Альтернативы**: подключить Cloudflare WAF Rate Limiting rule как внешний guardrail.

### H2. Возможная SSRF через INDEXNOW_ENDPOINT
1) **Где**: `functions/api/admin/articles.ts`.
2) **Тип**: Security (SSRF / egress abuse).
3) **Почему**: endpoint берется из env без allowlist/валидации и вызывается fetch-ом из backend.
4) **Прод-проявление**: при компрометации env/деплоя — запросы к внутренним/нежелательным хостам.
5) **Условия**: misconfig/secret leak.
6) **Критичность**: **HIGH**.
7) **Цепочка**: подмена env -> SSRF вызовы -> разведка внутренней сети/abuse -> инцидент.
8) **Воспроизведение**: выставить `INDEXNOW_ENDPOINT` на тестовый internal endpoint и сохранить статьи.
9) **Исправление**: жесткий allowlist доменов (`api.indexnow.org` и др.), валидация URL scheme/host/port.
10) **Альтернативы**: полностью убрать кастомный endpoint из env.

### H3. Ложноположительный успех отправки лидов
1) **Где**: `src/app/components/ContactForm.tsx`.
2) **Тип**: Product reliability / analytics integrity.
3) **Почему**: `mode: 'no-cors'` не дает проверить статус ответа; UI всегда трактует отправку как успешную.
4) **Прод-проявление**: потерянные лиды + искаженные метрики (события лидов отправляются даже при сбое).
5) **Условия**: 4xx/5xx у Google Script, сетевые ошибки partial.
6) **Критичность**: **HIGH**.
7) **Цепочка**: сбой интеграции -> UI “успех” -> пользователь уходит -> лид потерян -> неверная оптимизация маркетинга.
8) **Воспроизведение**: отключить endpoint/вернуть 500, отправить форму.
9) **Исправление**: серверный proxy endpoint с валидацией + проверка статуса + retry + idempotency key.
10) **Альтернативы**: заменить transport на CORS-enabled backend webhook.

## MEDIUM

### M1. Утечка внутренних ошибок в HTTP заголовок
1) **Где**: `functions/sitemap.xml.ts`, `functions/feed.xml.ts`.
2) **Тип**: Security/observability anti-pattern.
3) **Почему**: в `X-Error-Message` уходит текст исключения наружу.
4) **Прод-проявление**: информационная утечка о внутренних зависимостях/состоянии.
5) **Условия**: сбой JSONBin/fetch/render.
6) **Критичность**: **MEDIUM**.
7) **Цепочка**: ошибка -> раскрытие деталей -> упрощение reconnaissance -> усиление атак.
8) **Воспроизведение**: сломать upstream (JSONBin), запросить `/sitemap.xml`.
9) **Исправление**: убрать подробности из клиентского ответа, логировать только server-side.
10) **Альтернативы**: standardized error codes без деталей.

### M2. Нет верхних ограничений на размер payload статей
1) **Где**: `functions/api/admin/articles.ts`, `functions/_lib/jsonbin.ts`.
2) **Тип**: Performance/DoS.
3) **Почему**: нет лимитов на число статей, размер content, количество faq/tag.
4) **Прод-проявление**: рост latency/memory, дорогие кеш-инвалидации, риск упора в лимиты runtime.
5) **Условия**: случайно большой импорт или злоумышленник с паролем.
6) **Критичность**: **MEDIUM**.
7) **Цепочка**: oversized PUT -> тяжелая нормализация/запись -> timeouts -> stale content.
8) **Воспроизведение**: отправить PUT с очень большим `articles[]`.
9) **Исправление**: request size guard + schema validation (zod), hard caps.
10) **Альтернативы**: chunked update API.

### M3. Нестрогая supply-chain воспроизводимость
1) **Где**: `package.json`.
2) **Тип**: DevOps/Security.
3) **Почему**: нет lockfile в репозитории + диапазоны версий (`^`) для части deps.
4) **Прод-проявление**: nondeterministic builds, непредсказуемые регрессии/вулны.
5) **Условия**: новый install в CI/локально.
6) **Критичность**: **MEDIUM**.
7) **Цепочка**: свежая минорная версия -> изменение поведения -> инцидент в проде.
8) **Воспроизведение**: сравнить `npm i` в разное время на чистом окружении.
9) **Исправление**: commit lockfile + CI с `npm ci` + регулярный audit pipeline.
10) **Альтернативы**: pnpm + frozen lockfile.

### M4. Публичный fallback-контент как источник правды при сбоях
1) **Где**: `src/app/config.ts`, `src/app/components/hooks/useArticlesApi.ts`, `scripts/config.js`.
2) **Тип**: Data integrity/SEO consistency.
3) **Почему**: клиент при сбоях тянет публичный JSONBin URL и мешает источники данных.
4) **Прод-проявление**: расхождение контента между bot/html/api/spa, нестабильные SERP snippets.
5) **Условия**: частичный outage API/функций.
6) **Критичность**: **MEDIUM**.
7) **Цепочка**: primary fail -> fallback merge -> другой контент -> индекс/кэш рассинхрон.
8) **Воспроизведение**: уронить `/api/articles`, оставить JSONBin доступным.
9) **Исправление**: единый источник на edge API, deterministic fallback order + content hash/version.
10) **Альтернативы**: read-only signed snapshot в CDN.

## Quick wins (1-3 дня)
- Внедрить DOMPurify (write+read path), отключить raw HTML по умолчанию.
- Убрать `X-Error-Message` из публичных ответов.
- Перенести форму лидов на серверный endpoint с нормальными статусами.
- Добавить schema validation и лимиты payload для `/api/admin/articles`.
- Добавить lockfile + `npm ci` + `npm audit --production` в CI.

## Глубокие архитектурные улучшения (1-3 спринта)
- Перейти на структурированный rich-text формат (AST) вместо хранения HTML.
- Вынести rate-limit и auth hardening в edge perimeter (WAF/DO).
- Ввести content versioning и единый publish pipeline (draft->validate->publish).
- Добавить SLO/SLI, централизованные логи ошибок, бизнес-алерты по лидам.

## Риски масштабирования x10/x100
- Текущий rate-limit и кэш не обеспечат надежного backpressure.
- При массовом обновлении контента invalidate множества URL может стать bottleneck.
- Без жесткого контроля контента возрастает риск SEO-штрафов из-за inconsistent HTML/metadata.

## SEO/AEO/GEO эффект от предложений

Короткий ответ: **да, SEO/AEO/GEO должно заметно улучшиться**, если внедрить рекомендации без побочных regressions.

Ожидаемый эффект по направлениям:
- **Индексация и trust**: устранение XSS/небезопасного HTML снижает риск security-flags и проблем с качеством страниц у поисковых роботов.
- **Стабильность сниппетов**: единый источник контента и детерминированный fallback уменьшат расхождения между бот-версией и SPA.
- **Актуальность**: надежный publish/invalidate контур ускорит попадание обновлений в sitemap/feed и индексацию.
- **AEO**: структурированный FAQ/summary и чистая schema-разметка улучшают шанс попадания в ответные блоки.
- **GEO (generative engines)**: согласованный контент без рассинхрона и безопасный HTML повышают вероятность корректных цитат в генеративных ответах.

Примерно по приоритету влияния на SEO:
1. Закрыть XSS/unsafe HTML.
2. Стабилизировать источник данных и fallback.
3. Убрать публичные утечки ошибок и улучшить надежность API under load.
4. Укрепить release/CI-процессы, чтобы не ломать SEO регрессиями.

Важно: эффект не мгновенный. Обычно первые сигналы видны через 1-3 недели после релиза и переобхода страниц роботами.
