## Найденные проблемы

### 1) Критичная логика админки: риск тихой перезаписи статьи при конфликте slug (исправлено)
**Симптом:**
- При редактировании статьи и установке slug, который уже существует у другой статьи, код мог перезаписать чужую статью без отдельного подтверждения.

**Что сделано:**
- Добавлена защита: при конфликте slug сохранение блокируется с понятным alert.

**Результат:**
- Риск потери/перезаписи контента из админки устранён.

### 2) Средний риск: SEO/контент на этапе build зависит от fallback-цепочки
**Наблюдение:**
- При недоступности JSONBin сборка берет fallback-данные (предыдущий build cache / local).
- Это правильно для отказоустойчивости, но потенциально может отдать не самый свежий контент в статическом пререндере.

**Рекомендация:**
- Для прод-сборок включать stricter CI-политику (уже поддерживается переменными `REQUIRE_FRESH_ARTICLES`, `ALLOW_FALLBACK_BUILD`).

### 3) Низкий риск: состояние источника контента в health endpoint агрегированное
**Наблюдение:**
- `/api/health/content` возвращает `content_available` или `empty_or_fallback`, но без детального источника (primary/backup/seed).

**Рекомендация:**
- Добавить более granular `source` для оперативной диагностики в админке.
