## 0) Повторная проверка кейса «новые статьи не отображаются»

### Что можно улучшить
Сделать обновление контента более предсказуемым после сохранения из админки.

### Почему важно
Если после сохранения контент не виден сразу, теряется доверие к админке, замедляется публикация и страдает SEO-операционка.

### Что будет, если внедрить
Сократится доля «ложных инцидентов» (статья сохранена, но не видна), команда быстрее публикует и проверяет контент.

### Риски
Неверная стратегия инвалидации кэша может увеличить нагрузку на origin/functions.

### Как внедрить безопасно
1. Добавить в ответ admin PUT служебные поля: `cacheInvalidationAttempted`, `siteUrlUsed`, `invalidatedPathsCount`.
2. Логировать домен запроса и домен, по которому делается cache delete.
3. Добавить кнопку в админке «Принудительно обновить статьи» (manual refetch + bypass cache query).

### Гипотеза первопричины
- После сохранения вызывается cache invalidation на URL, построенных от `SITE_URL` или `request.url origin`.
- Если фактический домен пользователей отличается от `SITE_URL`, invalidate может уходить не по тому origin и пользователи продолжают видеть старый cache.

## 🚀 Быстрые улучшения (Quick Wins)

### QW-1. Добавить «статус источника данных» в UI админки
- Что улучшить: индикатор `JSONBin / fallback / local` рядом со списком статей.
- Почему: оператор сразу понимает, откуда загружены данные.
- Эффект: быстрее диагностика «почему не видно статьи».
- Риск: минимальный.
- Безопасно: read-only label, без влияния на контент.
- ДО/ПОСЛЕ (идея):
  - ДО: только список статей.
  - ПОСЛЕ: бейдж `Source: jsonbin` / `Source: fallback`.

### QW-2. Добавить в Admin кнопку «Обновить из API (no-cache)»
- Что улучшить: ручной контроль refetch после save.
- Почему: снимает зависимость от edge stale window.
- Эффект: быстрее подтверждение публикации.
- Риск: низкий.
- Безопасно: отдельная кнопка, не меняет дизайн страницы радикально.

### QW-3. Добавить health endpoint `/api/health/content`
- Что улучшить: observability (`articlesCount`, `updatedAtMax`, `source`).
- Почему: мониторинг в 1 запрос.
- Эффект: раннее обнаружение деградации.
- Риск: низкий.
- Безопасно: no-store, без приватных данных.

## 💰 Улучшения, влияющие на конверсию (CRO)

### CRO-1. Усилить CTA-путь без смены текстов
- Что улучшить: сделать CTA-кнопки семантическими `<button>/<a>` с корректным focus-visible.
- Почему: рост доступности = рост завершения действий на мобиле/клавиатуре.
- Эффект: меньше friction, выше micro-conversion.
- Риск: низкий.
- Безопасно: сохранить текущие классы и анимации.
- ДО/ПОСЛЕ:
  - ДО: кликабельные `div`.
  - ПОСЛЕ: `button` с теми же стилями.

### CRO-2. Добавить post-submit confidence signal
- Что улучшить: на ThankYou добавить проверяемый индикатор «заявка принята» + timestamp (без изменения смысла текста).
- Почему: снижает тревожность после отправки.
- Эффект: выше доверие, ниже повторные отправки.
- Риск: низкий.
- Безопасно: микро-блок без изменения визуального стиля.

## ⚡ Улучшения скорости

### PERF-1. Отложенная загрузка маркетинговых пикселей до явного user action
- Что улучшить: Meta/TikTok подгружать только при marketing consent + первом взаимодействии.
- Почему: быстрее TTI/INP на первом экране.
- Эффект: лучше web vitals.
- Риск: средний (нужно корректно задокументировать атрибуцию).
- Безопасно: feature-flag + сравнение метрик до/после.

### PERF-2. Стабилизировать горизонтальный скролл блога
- Что улучшить: ограничить wheel-intercept только когда контейнер под курсором и есть overflow.
- Почему: меньше UX-«залипаний» scroll.
- Эффект: лучше perceived performance.
- Риск: низкий.
- Безопасно: condition в обработчике, без редизайна.

### PERF-3. Прелоад только критичных ассетов
- Что улучшить: пересмотреть preload list и оставить только LCP-кандидатов.
- Почему: снижение конкуренции за сеть.
- Эффект: быстрее первый meaningful paint.
- Риск: низкий.
- Безопасно: A/B на preview окружении.

- Риск: низкий.
- Безопасно: debug-only метаданные.
